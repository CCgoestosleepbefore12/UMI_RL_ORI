# UMI-RL 方法选型分析

> 目的:为"在 UMI 预训练 flow-matching 策略上做 RL 后训练"确定方法路线,并记录**为什么选 / 为什么不选**的理由,供日后回看与他人复核。
> 关联:[[强化学习后训练 主题地图]]、`docs/CONTEXT.md`(术语与三家族)、`concepts/Advantage-conditioned 微调.md`。

---

## 1. TL;DR 结论

**真正的瓶颈不是"选哪个 RL 算法",而是"reward/value 信号从哪来"——三家族无一例外都依赖它。** 因此:

- **Phase 0(关键路径)**:先定 + 建 reward/value 信号。不解决它,后面全卡。
- **Phase 1(第一条 RL 线)**:**advantage-conditioned 微调**。样本效率最高、工程风险最低、最契合"真机只有几十条 + 工程优先"。
- **Phase 2(触顶再上)**:Q-chunking critic + QAM,更充分榨干稀缺在线数据。
- **备选**:世界模型想象(RISE 近亲),仅当真机数据连 Phase 1 都不够时启用。

一句话:**advantage-conditioned 是第一线而非 fallback**;Q-chunking + QAM 是升级而非起点。

---

## 2. 约束盘点(决策依据)

| # | 约束 | 取值 | 对选型的影响 |
|:--|:--|:--|:--|
| C1 | 预训练策略架构 | **flow-matching** | action log-likelihood 不可解 → 砍掉所有依赖 token-likelihood 的方法 |
| C2 | 真机交互预算 | **少量(几十条)** | 砍掉吃数据的纯在线 RL;偏向离线 + 少量在线、或把数据摊进模型 |
| C3 | 仿真器 | **不确定** | sim-based 方法暂不可依赖 |
| C4 | reward / 进度信号 | **未定** | 这是关键路径(见 §1);Phase 0 必须先解决 |
| C5 | 项目定位 | **工程优先,论文最后** | 目标函数 = 最稳、最可能真跑出提升;新颖性不是约束 |

> C4 是空的,且被 C1–C3 进一步收紧:flow + 几十条 + 无 sim,意味着 reward 既不能靠 sim 给、也不能靠海量在线采样自然涌现,只能**主动设计/学习**一个打分器。

## 3. 候选方法全景:三家族

flow 策略 RL 的核心矛盾——`log π(a|s)` 不可解,标准 PPO/GRPO 的 importance ratio 算不出来。绕过它的三条路:

| 家族 | 怎么绕过 likelihood | 代表 | 天然信号来源 | arXiv |
|:--|:--|:--|:--|:--|
| **(A) 去噪当 MDP/SDE** | 把逐步去噪 reparameterize 成 SDE,每步做 policy gradient | π_RL(Flow-SDE)、RL-100 | 在线 / 仿真 | 2510.25889 / 2510.14830 |
| **(B) critic 梯度蒸馏** | 把 critic 梯度经 adjoint matching 转成对 flow 每步的逐步监督 | QAM、LWD | offline-to-online | 2601.14234 |
| **(C) advantage-conditioned 监督** | 把 advantage 当条件,退化成监督式 BC,彻底躲开 likelihood | RECAP/RISE | 离线 / 世界模型想象 | 2511.14759 |

> 配套:Q-chunking(2507.07969)不是独立家族,而是**把 RL 放到 chunk 动作空间**的"地基"——它学的是 chunk-level critic,可以喂给 (B) 或 (C) 做策略改进。

## 4. 选型矩阵(约束 × 方法适配)

打分:✅ 适配 / ⚠️ 有条件 / ❌ 不适配。

| 方法 | C1 flow | C2 几十条 | C3 无 sim | C4 reward 未定 | C5 工程优先 | 综合 |
|:--|:-:|:-:|:-:|:-:|:-:|:--|
| **advantage-conditioned (C)** | ✅ 监督,不碰 likelihood | ✅ 纯离线+少量在线最省 | ✅ 不需要 sim | ⚠️ 需 value 打分器 | ✅ 风险最低、零件齐 | **第一线** |
| **Q-chunking + QAM (B)** | ✅ QAM 专治 flow | ⚠️ 几十条够 TD 但紧 | ✅ 不需要 sim | ⚠️ 需 critic | ⚠️ critic 穿 flow 有工程风险 | **升级** |
| **世界模型想象 (A/C)** | ✅ | ✅ 摊薄稀缺数据 | ✅ 自带"想象" | ⚠️ 想象中 reward 更脆 | ❌ 建模本身高风险 | **备选** |
| π_RL 纯在线 (A) | ✅ | ❌ 吃在线数据 | ⚠️ | ⚠️ | ⚠️ | 暂不 |
| token-GRPO(SimpleVLA/TGRPO) | ❌ likelihood 不可解 | — | — | — | — | **否决** |
| 纯在线 PPO/SAC | ⚠️ 需特殊处理 flow | ❌ 吃数据 | ❌ 常需 sim | ❌ | ❌ | **否决** |
| sim-based GRPO(RLinf-VLA 等) | ✅ | ✅ | ❌ sim 不确定 | — | ⚠️ | 暂不 |

**关键读法**:没有任何一列方法能绕开 C4。这印证了 §1——**reward/value 信号是先决条件,不是某个算法的附属选项**。

## 5. 推荐路线:分阶段 + 为什么选

### Phase 0 — reward/value 信号(关键路径)

**做什么**:在四类信号里定一个(或组合),并实现打分器:
1. progress 模型(从 demos 学单调进度)
2. 学习式 value/critic(demos + 几十条 rollout)
3. 稀疏成功 + leave-one-out advantage(RIPT-VLA 式,2505.17016)
4. HIL 人工打分/接管(ConRFT/VLAC 式,2502.05450 / 2509.15937)

**为什么先做它**:§4 矩阵显示三家族全部依赖它;它是唯一一个"不解决就没有下一步"的环节。这一步应单独 /grill + 设计。

### Phase 1 — advantage-conditioned 微调(第一条 RL 线)

**做什么**:用 Phase 0 的打分器给 demos + 几十条 rollout 标 advantage → 把 advantage 当条件输入,监督式微调 flow 策略,学"advantage 高时该怎么动"。

**为什么选**:
- **C1**:纯监督,根本不碰 log-likelihood,天然绕开 flow 的核心难点。
- **C2**:不需要在线探索,几十条 + demos 即可,样本效率三家族最高。
- **C5**:工程风险最低——没有"梯度穿过多步去噪"的不稳定性;且已有 `concepts/Advantage-conditioned 微调.md` 笔记 + 读过 RISE,认知零件最齐。
- 它能最快把"一个能用的 reward 信号"转化成"一个真的更强的策略",验证整条数据流是否成立。

### Phase 2 — Q-chunking critic + QAM(触顶再上)

**做什么**:在 chunk 动作空间学 TD critic(Q-chunking,2507.07969),用 QAM(2601.14234)把 critic 梯度蒸馏成对 flow 策略的逐步监督。

**为什么是升级而非起点**:advantage-conditioned 只用到 advantage 的"排序"信息,没充分利用 bootstrapping;当它触顶、而你又想从那几十条在线数据里榨出更多时,引入 TD + critic 梯度是合理的下一步。但"chunk-level critic + flow 策略 + QAM"是**未经广泛验证的新组合**,工程风险高于 Phase 1,故不作为第一线。

### 备选 — 世界模型想象(RISE 近亲)

**何时启用**:若真机数据连 Phase 1 都喂不饱。训一个 dynamics + value,把稀缺 rollout 摊进模型里无限想象 rollout(RISE / WMPO 2511.09515 / VLA-RFT 2510.00406 / DiWA 2508.03645)。

**为什么是备选**:它能摊薄稀缺真机数据,但建一个够准的世界模型本身是大工程,且想象空间里的 reward/value 更脆(§ 主题地图 §4 已标此风险)。与 C5"工程优先、最可能跑通"相悖,故列备选。

## 6. 为什么不选(逐个否决)

| 被否决项 | 否决理由 | 触发哪条约束 |
|:--|:--|:--|
| **token-GRPO**(SimpleVLA-RL / TGRPO / RIPT-VLA 的 GRPO 部分) | 前提是 token-level likelihood 可解,flow 策略不满足,直接失效 | C1 |
| **纯在线 PPO / SAC** | 吃大量在线交互;你只有几十条,且常依赖 sim | C2 / C3 |
| **Q-chunking 作为第一线**(他人推荐的"第一优先") | Q-chunking 价值在于对"有意义的在线交互"做 TD backup;几十条数据撑不起,按其自身决策树也应先走 advantage-conditioned | C2 |
| **直接上世界模型当第一线** | 建模脆、工程量大,与"最稳、最可能跑通"相悖;降为备选 | C5 |
| **sim-based GRPO**(RLinf-VLA 等框架路线) | 依赖可用 sim,而 sim 还不确定 | C3 |
| **π_RL 纯在线 Flow-SDE 当第一线** | 架构对口(✅ C1),但本质是在线 RL,吃数据;几十条不够,留作有在线预算后的选项 | C2 |

> 对"他人推荐"的总评:其三家族判断与本文一致,但(1)把"Q-chunking 第一优先"这个**条件结论**当成了已定决策,而你的"几十条"恰恰落在它自己写的"低预算→advantage-conditioned"分支;(2)它优化的是"最顺地跑起来",这与你"工程优先"一致,故本文采纳其大方向,只是按你的真实数字把**第一线从 Q-chunking 改为 advantage-conditioned**,并把 reward 信号提到 Phase 0。

## 7. 关键未决 + 下一步

- [ ] **Phase 0 是下一个该 /grill 的真问题**:reward/value 信号到底用哪种(progress / 学习式 value / 稀疏+LOO / HIL)、怎么从 demos + 几十条 rollout 学、怎么验证它可靠。**它比"选哪个算法"更决定成败。**
