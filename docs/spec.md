# UMI-RL 需求规格(spec)

> 方法路线见 `docs/方法选型分析.md`,关键决策见 `docs/adr/0001-先HG-DAgger再触发式RL.md`。
> 本 spec 聚焦 **Phase 0:HG-DAgger 回路**。Phase 1+(RL)待触发后另写。

## 0. 一句话

在固定窄场景下,用 **HG-DAgger**(人 gated 实时接管纠正 + BC 微调 flow 策略)把"协变量漂移/无法恢复"(b)做稳。**不解决广泛泛化 (a)**,**不训 reward model**(推迟到 Phase 1 触发)。

## 1. 目标与范围

- **目标**:窄场景下,策略从"漂出 demo 分布就回不来"→"能稳定完成接书→上架,且能从偏差中恢复"。
- **成功判据(待 /plan 量化)**:自主成功率曲线 + 人均接管率随 DAgger 轮次下降。
- **范围内**:窄场景定义、HIL 接管回路、数据聚合、flow 策略重训/微调、迭代收敛判据。
- **范围外**:(a) 广泛泛化(留给 UMI 多样数据 + 表征)、reward model、任何 RL。

## 2. 底座与约束

- 预训练策略:flow-matching(动作维度 / chunk 长度 / 网络:**待确认**)。
- 硬件:机械臂 + 遥操实时接管(HIL-SERL 就绪)。
- 观测模态:RGB / 腕部相机 / 本体位姿 / 夹爪宽度 —— **待 UMI obs 定**。
- 真机预算:少量(几十条量级,可多轮)。

## 3. HG-DAgger 回路(核心,细节待 /plan)

- [ ] **窄场景定义**:房间 / 书目 / 书架位置 / 手势范围的具体边界。
- [ ] **接管协议**:人何时介入(gated)、介入/退出的交互方式、接管段如何标注。
- [ ] **数据聚合**:demo + 纠正段如何合并;是否对纠正段加权;是否保留纯自主失败段(默认丢,见 ADR)。
- [ ] **重训**:从预训练权重微调 vs 重训;用原 flow-matching loss;学习率 / 轮次 / 是否冻结部分层。
- [ ] **迭代**:每轮 rollout 数、停止判据(成功率/接管率曲线)。

## 4. 模块映射(umi_rl/)

- `rollout/`:真机 rollout + HIL 接管采集
- `policy/`:加载/微调 UMI flow 策略(尽量当库用)
- (Phase 1 才用)`reward/` `critic/` `algo/`

## 5. 验证与评估

- [ ] 自主成功率(人不介入)随轮次
- [ ] 人均接管率随轮次
- [ ] 恢复能力:人为制造偏差,看能否回到成功轨迹

## 6. Phase 1 触发判据(占位,待量化)

- 见 ADR 0001:脱离人需求 / BC 触顶 / 超过 teleop。具体阈值待 /plan 定。
